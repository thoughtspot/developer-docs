  "content": "= REST API v2.0 changelog\n:toc: true\n:toclevels: 1\n\n:page-title: REST API v2.0 changelog\n:page-pageid: rest-v2-changelog\n:page-description: Changelog of REST APIs\n\nThis changelog lists the features and enhancements introduced in REST API v2.0. For information about new features and enhancements available for embedded analytics, see xref:whats-new.adoc[What's New].\n\n== Version 26.8.0.cl, August 2026\n\n=== Object identity and TML APIs\n\nObject ID support for Collections::\n// SOURCE: SCAL-317357\nCollections now support stable object IDs in TML import, export, and the metadata update API.\n\n* The `POST /api/rest/2.0/metadata/headers/update_obj_id` endpoint now accepts `COLLECTION` as a valid value for the `metadata_type` parameter. This allows you to assign or update stable object IDs for Collections programmatically, enabling consistent SDLC promotion across environments.\n\n* The TML import and export APIs now accept `COLLECTION` as a supported metadata type:\n** +++<a href=\"{{navprefix}}/restV2-playground?apiResourceId=http%2Fapi-endpoints%2Fmetadata%2Fimport-metadata-tml\">POST /api/rest/2.0/metadata/tml/import</a>+++ \u2014 set `metadata_type` to `COLLECTION` to import Collection TML files.\n** +++<a href=\"{{navprefix}}/restV2-playground?apiResourceId=http%2Fapi-endpoints%2Fmetadata%2Fexport-metadata-tml\">POST /api/rest/2.0/metadata/tml/export</a>+++ \u2014 set `metadata_type` to `COLLECTION` to export Collections as TML.\n\nFor more information, see xref:tml.adoc[TML].\n\nPersonalized Views in TML::\n// SOURCE: SCAL-307283\nThe TML schema for Liveboards now includes `owned_by` and `obj_id` fields for Personalized Views. The TML import API applies smart-merge behavior for views:\n\n* Views that exist only in the destination environment are preserved.\n* Views that exist only in the imported TML are appended.\n* Views that exist in both the imported TML and the destination environment are updated.\n\nThe `PERSONALIZED_VIEW` type is also supported in the `POST /api/rest/2.0/metadata/headers/update_obj_id` endpoint, allowing you to assign stable object IDs to Personalized Views programmatically.\n\n// NOTE TO WRITER: Verify exact TML YAML field names (`owned_by`, `obj_id`) and their placement within the Liveboard TML structure with the TML/Liveboard team before publishing. (SCAL-307283)\n\nFor more information, see xref:tml.adoc[TML].\n\n=== User management and access control\n\nGranular download privileges::\n// SOURCE: SCAL-273333\nThe `Can Download Data` privilege has been split into two separate privileges. Update REST API calls that assign or query privileges for user groups or roles to use the new values:\n\n* `DATADOWNLOADING_VISUALS` \u2014 grants permission to download Liveboards and visualizations as PDF or PNG.\n* `DATADOWNLOADING_DETAILED` \u2014 grants permission to download raw tabular data as XLSX or CSV.\n\n// NOTE TO WRITER: Verify the exact REST API v2 enum strings for both new privilege values with the Platform/IAM team before publishing. (SCAL-273333)\n\n=== Scheduled jobs\n\nDeprecation of per-minute schedule frequency::\n// SOURCE: SCAL-244749\nThe `MINUTES` frequency type is no longer supported in Liveboard notification schedule requests. Remove `MINUTES` from the frequency type in any API calls to the schedule create or update endpoints. The supported frequency types remain `HOURLY`, `DAILY`, `WEEKLY`, and `MONTHLY`.\n\n== Version 26.7.0.cl, July 2026\n\n=== Spotter AI APIs\n\nSave chat AI APIs::\nThoughtSpot introduces the following REST API endpoints and enhancements to manage saved Spotter conversations programmatically. These endpoints allow you to build custom conversation history interfaces in embedded applications without using the native Spotter UI.\n\n* `POST /api/rest/2.0/ai/agent/conversations/{conversation_identifier}/update` +\nUpdates attributes of an existing agent conversation.\n* `GET /api/rest/2.0/ai/agent/conversations` +\nRetrieves the list of saved agent conversations for the currently authenticated user.\n* `GET /api/rest/2.0/ai/agent/conversations/{conversation_identifier}/messages` +\nRetrieves the full content of a saved conversation with Spotter agent.\n* `POST /api/rest/2.0/ai/agent/conversations/{conversation_identifier}/update` +\nUpdates the display title of a saved conversation.\n* `DELETE /api/rest/2.0/ai/agent/conversations/{conversation_identifier}/delete` +\nDeletes a saved conversation and all its associated messages.\n* `GET /api/rest/2.0/ai/agent/conversations/{conversation_identifier}/answers/{answer_identifier}/details` +\nLoads the full answer payload for a specific answer item in an agent conversation.\n\n+\nFor more information, see xref:spotter-agent-conversation-mgmt-apis.adoc[Saving and managing Spotter AI chat].\n\nSave chat settings in conversation create API::\n\nThe `POST /api/rest/2.0/ai/agent/conversation/create` API endpoint is modified to allow users to save conversations by setting `enable_save_chat: true` in the API request.\n\nAgent instructions APIs::\nThe following new API endpoints allow you to set and retrieve persistent behavioral instructions for the Spotter agent.\n\n* `PUT /api/rest/2.0/ai/agent/instructions/set` +\nSets behavioral instructions for the Spotter agent. Use this endpoint to define persistent guidance that Spotter applies when responding to queries in a conversation session.\n\n* `GET /api/rest/2.0/ai/agent/instructions/get` +\nRetrieves the behavioral instructions currently configured by the administrator for the Spotter agent.\n+\nFor more information, see xref:spotter-agent-instructions.adoc[Spotter AI Agent instructions APIs].\n\n=== Webhooks\n\nNew API endpoint::\nThe `GET /api/rest/2.0/webhooks/storage-config` API endpoint allows retrieving the storage setup information required for configuring a GCS or S3 storage destination for webhook delivery.\n\nEnhancements::\n\n* The `/api/rest/2.0/webhooks/create` endpoint allows activating a webhook and configuring GCS storage destination for webhook delivery. The API endpoint also returns GCS storage configuration details in the response.\n* The `/api/rest/2.0/webhooks/{webhook_identifier}/update` API endpoint supports configuring webhook activate status, resetting authentication, signature verification, and storage destination properties.\n* API requests to the `/api/rest/2.0/system/communication-channels/validate` now return GCS storage properties in response.\n\n=== Communication channel monitoring\n\nThe `/api/rest/2.0/jobs/history/communication-channels/search` API endpoint supports the `end_epoch_time_in_millis` parameter, which allows fetching records with a specific end timestamp.\n\n=== Style customization APIs\n\nThoughtSpot introduces the following REST API v2.0 endpoints to manage style customization settings programmatically.\n\nStyle configuration::\n\n* `POST /api/rest/2.0/customization/styles/search` +\nReturns the current style configuration at the `CLUSTER` or active `ORG` scope.\n\n* `POST /api/rest/2.0/customization/styles/update` +\nUpdates style settings at the `CLUSTER` or `ORG` scope.\n\nCustom fonts::\n\n* `POST /api/rest/2.0/customization/styles/fonts/upload` +\nUploads a custom font file to ThoughtSpot.\n\n* `POST /api/rest/2.0/customization/styles/fonts/search` +\nReturns custom fonts uploaded to the instance.\n\n* `PUT /api/rest/2.0/customization/styles/fonts/{font_identifier}/update` +\nUpdates the display name, weight, style, or color of an existing custom font.\n
