=== Support for Spotter 3 [.version-badge.new]#New#
ThoughtSpot MCP Server now supports Spotter 3, enabling advanced analytics, forecasting, multi-step reasoning, and deep research. The new MCP tools support real-time streaming, session-based conversations, richer interactions, and direct embedding of visualizations in MCP-aware apps. +
For more information, see xref:mcp-integration.adoc#_mcp_server_architecture_and_tools[MCP Server architecture and tools] documentation.

=== MCP tools and processing model [.version-badge.breaking]##Breaking change##
The new Spotter 3-powered MCP tools are available at a separate MCP Server URL. +
Existing integrations with legacy MCP tools continue to work until further notice. No tools from the previous version carry over into the new version. You must update your integrations to use the new MCP tools. Ensure that you test new MCP tools in staging before production rollout.

[cols="1,2,2", options="header"]
|====
|Legacy MCP tools |Replaced by|Notable changes

|`ping`
|`check_system_readiness`
|No change in functionality.

|`getDataSourceSuggestions`
|`data_source_id` parameter in `create_analysis_session`
|No longer a standalone tool. Is built into `create_analysis_session` via optional `data_source_id` parameter. Data source is selected automatically if omitted.

|`getRelevantQuestions`
|`send_session_message` and `get_session_updates` loop
|Split into asynchronous send message and poll pattern. Polls for streamed updates across multiple calls and accumulates updates from every poll.

|`getAnswer`
|`get_session_updates` loop
|Is now part of the polling loop.

|`createLiveboard`
|`create_dashboard`
|Creates a dashboard from `answer_id` values collected from the session update type `answer`. Requires a `note_tile` HTML summary tile.
|====

[IMPORTANT]
====
The `send_session_message` does not return an answer. Integrations must now implement an asynchronous polling loop via `get_session_updates` until `is_done: true`. For detailed information about tools and resources, see xref:mcp-tool-reference-guide.adoc[MCP tool reference guide].
====

=== MCP server URL changes and API versioning [.version-badge.breaking]##Breaking change##
All MCP Server URLs now support date-based versioning, defined using the `?api-version=YYYY-MM-DD` parameter. This enhancement introduces several notable changes for MCP clients using OAuth and bearer token authentication. Ensure that you update MCP Server URLs and authentication as needed.

[cols="6,4", options="header"]

|`\https://agent.thoughtspot.app/bearer/mcp` [.version-badge.deprecated]#Deprecated# +

a|**URL with the API version**:  +
`\https://agent.thoughtspot.app/token/mcp?api-version={YYYY-MM-DD}` +

`\https://agent.thoughtspot.app/token/mcp`. Always points to the latest version.

|`\https://agent.thoughtspot.app/mcp` +

|`\https://agent.thoughtspot.app/mcp?api-version={YYYY-MM-DD}` +

|`\https://agent.thoughtspot.app/openai/mcp` [.version-badge.deprecated]#Deprecated# +

| `\https://agent.thoughtspot.app/token/mcp?api-version={YYYY-MM-DD}`. +

|====

[IMPORTANT]
====
* The MCP Server URLs without the API version always point to the latest version and may introduce changes. Pin your integration to a tested version using `?api-version={YYYY-MM-DD}` parameter. For example, `?api-version=2026-04-30`.
* If the API version is set to a future date, the server returns the latest API version released before the date specified in the URL.

* To pin your integration to a specific version, append the `?api-version={YYYY-MM-DD}` parameter to the MCP Server URL and specify the date string.
====

=== Migration guidelines

Migrating from your existing setup to the new version requires updating the MCP client configuration to point to the new URL and rewriting tool-calling logic to use the new asynchronous, polling model. There is no change in the supported authentication models, MCP clients, or the MCP Server hosting domain.

To update your integration to use Spotter 3 features and the new MCP tools, follow these steps:

. Update your MCP Server URL to the latest version. Replace the old URLs in all MCP client configuration files.
. Rewrite your integration to use the new tools. Replace the old synchronous tool-calling pattern with the new asynchronous polling pattern. For code samples and tool reference, see the xref:mcp-tool-reference-guide.adoc[MCP tool reference guide].
. Test your integration end-to-end.
. Roll out to production environments after validation.

=== MCP tools, processing model, and answer retrieval
Supports the following MCP tools released in the initial version of ThoughtSpot MCP Server.

* `ping`: Tests connectivity and authentication.
* `getDataSourceSuggestions`: Returns available data source suggestions
for a given query.
* `getRelevantQuestions`: Breaks a query into sub-questions. Returns answers synchronously.
* `getAnswer`:  Returns the answer to a specific sub-question.
* `createLiveboard`: Creates a ThoughtSpot Liveboard from a list of answers.

==== MCP Server URLs
Supports the following URLs:

* `\https://agent.thoughtspot.app/mcp` (OAuth Apps)
* `\https://agent.thoughtspot.app/bearer/mcp` (Bearer Token Apps)
* `\https://agent.thoughtspot.app/openai/mcp` (OpenAI-compatible clients)

These URLs are deprecated in the MCP Server 2026-04-30 release. Refer to the xref:mcp-server-changelog.adoc#_mcp_server_url_changes_and_api_versioning[changelog] for more information.
