  "content": "= Cluster maintenance and upgrade\n:toc: true\n:toclevels: 1\n\n:page-title: Update cluster state\n:page-pageid: tse-cluster\n:page-description:  If you are using a ThoughtSpot Cloud cluster in the economy mode in your embedded deployments, use the APIs to restart an inactive cluster.\n\nAt any given time, a ThoughtSpot application instance can be in any one of the following states:\n\n* `ACTIVE` +\nIndicates that the cluster is active and user activity is detected.\n* `UNDER_MAINTENANCE` +\nIndicates that the cluster is down for maintenance due to upgrade or patching.\n* `STOPPED`\nIndicates that the cluster is stopped and no user activity is detected.\n* `STARTING`/`PENDING` +\nThe cluster is currently starting, or some other workflow is running on the cluster.\n\n== Cluster status during upgrade\nWith ThoughtSpot's Minimal Downtime Ephemeral Mode upgrade option, we upgrade ThoughtSpot in the background while users can use ThoughtSpot in Ephemeral mode. This means that during the upgrade, the system will be in transient state, yet it allows users to create and view data. However, any new objects created during the upgrade will be lost.\n\n// SOURCE: SCAL-320519\nWhen the upgrade starts, the ThoughtSpot instance operates in Ephemeral mode and the cluster state changes to `UNDER_MAINTENANCE`. In Ephemeral mode, users can continue to create and view data, but any new objects created during the upgrade are discarded when the upgrade completes.\n\n[NOTE]\n====\nRead-Only mode has been deprecated and is no longer available. The current upgrade experience uses Ephemeral mode only.\n====\n\nThoughtSpot users can determine if their instance is under maintenance by sending a `GET` request to one of the following API endpoints:\n\n* REST API v1 endpoint +\n`GET /tspublic/v1/admin/banner`\n* REST API v2 endpoint +\n`GET /api/rest/2.0/system/banner`\n\nThoughtSpot Embedded application users can also view the banner text by calling the above APIs.\n\n=== REST v1 API request\n\n.cURL\n[source,cURL]\n----\ncurl -X GET \\\n--header 'Accept: application/json' \\\n'https://{ThoughtSpot-Host}/callosum/v1/tspublic/v1/admin/banner'\n----\n\n.Request URL\n\n----\nhttps://{ThoughtSpot-Host}/callosum/v1/tspublic/v1/admin/banner\n----\n\n=== REST v2 API request\n\n.cURL\n[source,cURL]\n----\ncurl -X GET \\\n--header 'Accept: application/json' \\\n'https://{ThoughtSpot-Host}/api/rest/2.0/system/banner'\n----\n\n.Request URL\n----\nhttps://{ThoughtSpot-Host}/api/rest/2.0/system/banner\n----\n\n=== API response\n\nIf the cluster in maintenance mode, the API returns the following response:\n----\n{\"banner_text\":\"This system is currently being upgraded and is in ephemeral mode. You can continue to use it to visualize data. Any objects you create or modify during this period will be lost when the upgrade is complete.\",\"under_maintenance\":true}\n----\n\nIf the cluster is not in maintenance mode, the API returns the following response:\n----\n{\"banner_text\":\"This system is functioning normally. No maintenance is in progress.\",\"under_maintenance\":false}\n----\n\nAdministrators of the ThoughtSpot Embedded app can create their custom banner text and display it to indicate the cluster upgrade status.\n\n== Idle sensing\nIf you are using a ThoughtSpot Cloud instance for embedded deployments in your development or production environment, you can enable idle sensing to save costs and allow your cluster to operate in `economy` mode.\n\nIf idle sensing is enabled on your cluster, your cluster will be automatically stopped if there is no user activity detected for a given time threshold. By default, the idle time threshold is set to 120 minutes. To enable this feature on your clusters, contact ThoughtSpot Support.\n\n=== Get information about the status of a cluster\nBy default, a ThoughtSpot cluster running the `economy` mode stops if there is no user activity for two hours. When a user tries to access a cluster that's in the `STOPPED` state, the API calls will return the `\"cluster-state\": \"Stopped\"` in the response header.\n\n[source,cURL]\n----\n $ curl https://{cluster-name}.thoughtspot.cloud  --head\n  HTTP/1.1 200 OK\n  Server: awselb/2.0\n  Date: Tue, 30 May 2023 15:27:01 GMT\n  Content-Type: text/html\n  Content-Length: 3348\n  Connection: close\n  Cluster-State: Stopped\n----\n\nTo restart the cluster, complete the steps described in the following section.\n\n== Start an inactive cluster using API\nOn a regular ThoughtSpot Cloud cluster, users can restart an inactive cluster using `CAPTCHA`. However, on embedded instances, the `CAPTCHA`-based cluster activation is not supported. Instead, the embedded  application user can send a `GET` request to their instance with the following query parameters in the request URL:\n\n* `tse=true`\n* `start_cluster=true`\n\nFor example, to start an inactive cluster, send a `GET` request in the following parameters:\n\n.Production environment\n[source,http]\n----\nhttps://{ThoughtSpot-Host}/?tse=true&start_cluster=true\n----\n\n**Staging environment**\n\n[source,http]\n----\nhttps://{cluster-name}.thoughtspotstaging.cloud/?tse=true&start_cluster=true\n----\n\n**Development environment**\n\n[source,http]\n----\nhttps://{cluster-name}.thoughtspotdev.cloud/?tse=true&start_cluster=true\n----\n\nIn the request header, you must include the `security-key`. This `security-key` is used to authenticate your request when xref:trusted-auth-secret-key.adoc#trusted-auth-enable[trusted authentication is enabled]. ThoughtSpot Embedded users can obtain the `security key` for their instance or Org context from their ThoughtSpot administrator.\n\nIf your instance has Orgs:\n\n* Specify the Org ID in the request header along with the security key of that specific Org context. If the security key does not match the Org ID, the API returns an error.\n* If the request header includes only the security key with no Org ID, it's considered to be the security key of the global Org context (ALL Orgs). If it does not match the key generated for the All Orgs context, the API returns an error.\n\nWhen the cluster becomes available, users are logged into the Org context based on the key provided in the request.\n\nThe following example shows the cURL request for restarting a cluster:\n\n[source, cURL]\n----\n$ curl -X GET 'https://<cluster-name>.thoughtspot.cloud/?tse=true&start_cluster=true'  \\\n -H  'X-Thoughtspot-Org-Id: {Org_Id}' \\\n -H  'security-key: e8ade677-c3f1-461d-8b7f-7f0fe4e024f0'\n----\n\nIf the `GET` request is successful, the cluster starts.\n\n== Response header\nNote the cluster state in the response header:\n\n* `STARTING` +\nIndicates that the cluster is starting. It may take a few minutes for the cluster to become active.\n* `UNKNOWN` +\nIndicates a possible error. Contact your administrator or ThoughtSpot Support if the cluster does not start in 5-10 minutes.\n\n[source,]\n----\n HTTP/1.1 200 OK\n Server: awselb/2.0\n Date: Tue, 30 May 2023 16:04:08 GMT\n Content-Type: text/html\n Content-Length: 0\n Connection: keep-alive\n Cluster-State: Starting\n----\n\n== Response codes\n[options=\"header\", cols=\"1,4\"]\n|===\n|HTTP status code|Description\n|**200**|Successful operation\n|**400**|Invalid request\n|**401**|Unauthorized access\n|===\n",
