== August 2026

**Release version**: ThoughtSpot Cloud 26.8.0.cl +
*Upgrade notes*: See link:#_breaking_changes[Breaking changes] below before upgrading. +
*Recommended SDK versions*: Visual Embed SDK v1.51.0 and later

[#_breaking_changes]
[WARNING]
====
*26.8.0.cl contains three breaking changes for embedded applications:*

. *SDK flag defaults changed* — `hideIrrelevantChipsInFilterPanel`, `compactHeader`, `coverFilterPage`, and `aether` are now enabled (`true`) by default. Set them explicitly to `false` to preserve previous behavior.
. *Home page and navigation V1/V2 removed* — All embedded sessions now render the V3 experience. `HomePage.V1`, `HomePage.V2`, and the equivalent navigation flags are ignored by the platform. Migrate your `AppViewConfig` before upgrading.
. *Granular download privileges* — The *Can Download Data* privilege is split into *Can Download Visuals* (PDF/PNG) and *Can Download Detailed Data* (XLSX/CSV). Update any programmatic privilege assignments before upgrading.
====

==== SDK flag defaults updated [.version-badge.breaking]#Breaking#

The following SDK flags are now enabled (`true`) by default for all embedded ThoughtSpot applications in Visual Embed SDK v1.51.0. If your application explicitly passes `false` for any of these flags, that value is still honored. If you relied on the previous default of `false`, you must now explicitly set the flag to `false` to preserve that behavior.

[cols="2,4", options="header"]
|===
| Flag | Description
| `hideIrrelevantChipsInFilterPanel` | Hides filter chips that are not relevant to the currently displayed visualization. Previously disabled by default.
| `compactHeader` | Enables the compact Liveboard header layout. Previously disabled by default.
| `coverFilterPage` | Applies a cover layout to the Liveboard filter page. Previously disabled by default.
| `aether` | Enables the Aether visual design system for embedded components. Previously disabled by default.
|===

==== Home page and navigation V1/V2 removed [.version-badge.breaking]#Breaking#

In 26.8.0.cl, the classic (V1) and V2 home page and navigation experiences are removed from all ThoughtSpot embedded applications. All embedded sessions render the V3 navigation and home page experience regardless of the SDK configuration passed at initialization.

If your application uses any of the following, update your configuration before upgrading:

* `homePage: HomePage.V1` or `homePage: HomePage.V2` in `AppViewConfig`
* Navigation flags that previously enabled V1 or V2 navigation

*Migration steps:*

. Update `AppViewConfig` to use `discoveryExperience` with `primaryNavbarVersion: PrimaryNavbarVersion.Sliding` and `homePage: HomePage.ModularWithStylingChanges` for the V3 experience, or `homePage: HomePage.Focused` for the V4 focused home page.
. Remove any conditional logic in your codebase that branches on `HomePage.V1` or `HomePage.V2`.
. Review and test any CSS customizations targeting V1 or V2 navigation selectors.

For migration details, see xref:full-app-customize.adoc[Customize full application embedding] and xref:customize-homepage-full-embed.adoc[Customize home page experience].

==== Personalized Views — TML portability [.version-badge.new]#New#

Personalized Views in Liveboards can now be reliably ported across ThoughtSpot environments using TML. The following changes apply to Liveboard TML and the Import/Export TML APIs:

New TML fields:::
* `owned_by` — Assigns the owner of each Personalized View by `username` or email address. Preserves ownership when a Liveboard is promoted between clusters or Orgs.
* `obj_ID` — A stable identifier for each view across environments, enabling reliable merge behavior on import.

Smart merge on import:::
When importing a Liveboard TML containing Personalized Views, ThoughtSpot now:

* Preserves views that exist only in the destination environment.
* Appends views that exist only in the incoming TML.
* Updates views present in both the TML and the destination environment.

The previous behavior overwrote all destination views on import. This change ensures end-user Personalized Views are not deleted during Liveboard promotions.

==== Granular download privileges [.version-badge.breaking]#Breaking#

The *Can Download Data* privilege has been split into two separate privileges that provide more granular control over download permissions:

[cols="2,3", options="header"]
|===
| New privilege | Grants access to
| *Can Download Visuals* | PDF and PNG format downloads of Liveboards and visualizations
| *Can Download Detailed Data* | XLSX and CSV format downloads of Liveboards and visualizations
|===

*Impact on embedded applications:*

* Users who previously held the *Can Download Data* privilege are automatically migrated. Access is preserved without action.
* If your application makes programmatic privilege assignments via REST API or TML-defined role definitions, update those assignments to use the new granular privilege names before upgrading.
* If your application gates the visibility of download `Action` IDs based on privilege checks, validate your logic against the new privilege structure.

[discrete]
==== Collections object type support in TML and object ID APIs [.version-badge.new]#New#

ThoughtSpot Collections objects now support distributed object identity (`obj_id`) across the following areas:

TML import and export:::
The `COLLECTION` object type is now a supported enum value in the TML Import and Export APIs. You can export a Collection as a TML file, modify it, and import it back into ThoughtSpot — using the same `POST /api/rest/2.0/metadata/tml/import` and `POST /api/rest/2.0/metadata/tml/export` endpoints.

Update Object ID API:::
The `COLLECTION` object type is now supported in the Update Metadata Object ID API (`POST /api/rest/2.0/metadata/update_metadata_obj_id`). This allows you to assign and manage stable, portable object identifiers for Collections across ThoughtSpot environments.

Update Metadata Header API:::
The `COLLECTION` object type is now supported in the Update Metadata Header API. You can programmatically update the name and description of Collection objects.

This feature is available on ThoughtSpot Cloud 26.8.0.cl and ThoughtSpot Software.

For more information, see xref:tml.adoc[TML import and export] and xref:rest-api-v2-reference.adoc#_metadata[Metadata REST API reference].

[discrete]
==== Scheduled report attachment size limit correction [.version-badge.fixed]#Fixed#

Documentation correction: the maximum supported attachment size for scheduled Liveboard report emails is *22 MB*, not 25 MB as previously documented. The 22 MB limit accounts for base64 encoding overhead, which consumes approximately 33% of the raw file size. Files exceeding 22 MB cannot be delivered as email attachments and will cause the scheduled job to fail.
