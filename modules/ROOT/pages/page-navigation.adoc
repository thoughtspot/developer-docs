If you want to embed specific pages of the ThoughtSpot application, you can embed individual components such as `LiveboardEmbed`, `SpotterEmbed`, or `SearchEmbed` rather than full application embedding. The full application embed is a relatively fixed and unified application shell, while individual component embeds offer more granular and isolated control.

If you choose to embed individual components with specific pages, your embedding app may require custom menus, navigation elements with xref:customize-links.adoc[routes for pages], and ThoughtSpot REST API integration to programmatically manage interaction and navigation between these pages.

[NOTE]
====
Mixing full application embedding with other embed components, such as `SearchEmbed`, `SpotterEmbed`, and `LiveboardEmbed`, is not recommended, because each embed method is designed to manage its own navigation, layout, and application state. Combining them can create complicated workflows and make it harder to maintain and customize your embedded application.
====

See the link:https://github.com/thoughtspot/ts_everywhere_resources/blob/master/examples/menus_and_other_navigation_components/rest_api_ui_navigation_v2.html[code example on GitHub, window=_blank] for a complete implementation of REST API requests powering navigation components in an embedded application.

== Create menus to embedded ThoughtSpot pages
Once you have the URL routes defined within the embedding application to load ThoughtSpot content, you can write code that generates links to these ThoughtSpot routes.

// MINOR-2: dropdown → drop-down (WordList.yml)
Navigation elements often take the form of a sidebar menu, a drop-down selector, a set of tiles, or individual buttons.

The examples in the following sections use plain JavaScript to generate unstyled HTML elements.

The actual code for generating links and navigation elements will be written in your web application framework. Your code will use the same REST API requests and responses shown in the examples.

== Hardcoded links within a menu system
If your web application only embeds a limited set of ThoughtSpot objects, without granting users the ability to save their own objects, you can __hardcode__ the object IDs and names of the ThoughtSpot content.

Object IDs in ThoughtSpot are GUIDs and are unique on every Org or instance, except for separate instances that are intentionally replicated with the same GUIDs via TML import and export.

// MINOR-1: CICD → CI/CD (WordList.yml)
You can use the xref:rest-api-v2-metadata-search.adoc[`/api/rest/2.0/metadata/search`] V2.0 REST API endpoint to retrieve the equivalent object IDs for content on various Orgs and environments when building your hardcoded menu system. Alternatively, you can maintain a *mapping file* of equivalent objects during your CI/CD deployment process to generate the hardcoded URLs in the menu system.

== Dynamic requests for content listings via REST API
// MINOR-5: Removed "Both the V1 and V2.0 REST APIs" framing — V2.0 is the only recommended API
The xref:rest-api-v2-metadata-search.adoc[`/api/rest/2.0/metadata/search`] V2.0 REST API endpoint retrieves filtered lists of the content a user has access to.

The endpoint can be used by administrators to retrieve content for any given user, or called as the user directly from the web browser after signing in via SSO.

The response to `/api/rest/2.0/metadata/search` can be parsed and filtered on *tags*, *author*, or other properties. You can also make multiple calls to the endpoint with different filtering arguments in each request. For example, xref:rest-api-v2-metadata-search.adoc#_favorite_object_options[favorite options] can be specified to retrieve only objects that are marked as favorites by a user.

// MINOR-6: Removed "/metadata/list V1 REST API endpoint" paragraph — deprecated endpoint removed
// SOURCE: rest-api-v2-metadata-search.adoc: "This endpoint replaces metadata/list,
// metadata/listobjectheaders, metadata/details, and metadata/listvizheaders from REST API v1."

The process for generating a dynamic menu includes the following steps:

. Request the user's available content via the REST API.
. (Optional) Cache the results within the embedding application.
. Render menu items and links using the REST API response to build URLs within the web app that load ThoughtSpot content.

If implementing a caching system for ThoughtSpot content listings, handling content requests on the backend may be preferable for easier access to other parts of the web application.

=== Requesting content for a user
An administrator account can use the xref:rest-api-v2-metadata-search.adoc#permissions[permissions parameters] of the `/api/rest/2.0/metadata/search` V2.0 REST API endpoint to retrieve the list of content available for a specific user at either the `READ_ONLY` or `MODIFY` permission levels.

By default, the endpoint returns Liveboards. You can set the `metadata` array in the request body to specify `LIVEBOARD`, `ANSWER`, and `LOGICAL_TABLE` in any combination.

// CRIT-4: Fixed code block — [code,json] → [source,javascript], mixed quotes fixed,
// missing comma after permissions array added, consistent double-quoting throughout
[source,javascript]
----
const answerRequestObject = {
    "metadata": [{ "type": "ANSWER" }],
    "permissions": [
        {
            "principal": {
                "type": "USER",
                "identifier": "{username}"
            },
            "share_mode": "READ_ONLY"
        }
    ],
    "record_offset": 0,
    "record_size": 10000
};

const response = await fetch('/api/rest/2.0/metadata/search', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        'Accept': 'application/json',
        'Authorization': 'Bearer {access-token}'
    },
    body: JSON.stringify(answerRequestObject)
});

const contentList = await response.json(); // Returns a top-level array
----

If the web application uses a ThoughtSpot service account with administrator privileges on the backend, this approach is simpler than managing individual user access tokens for each request.

=== Requesting content as a user
Rather than using an administrator account to request content listings, you can scope the REST API request to the signed-in user directly. The REST API then returns only the content that user has access to.

All REST API requests from the browser are scoped as the signed-in user, as long as the `credentials` option of the fetch request is set to `include`.

// CRIT-5: Fixed code block — [code,json] → [source,javascript], single-quoted keys fixed,
// consistent double-quoting throughout
[source,javascript]
----
const lbRequestObject = {
    "metadata": [{ "type": "LIVEBOARD" }],
    "record_offset": 0,
    "record_size": 10000
};

const response = await fetch('/api/rest/2.0/metadata/search', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        'Accept': 'application/json'
    },
    body: JSON.stringify(lbRequestObject),
    credentials: 'include'
});

const contentList = await response.json(); // Returns a top-level array
----

With xref:trusted-authentication.adoc[cookieless trusted authentication], there is no browser session. Instead, an access token is retrieved for the user and used by the SDK.

The trusted authentication pattern requires implementing a backend service to generate access tokens for any user. This token service can also be used by the backend of the embedding web application to make REST API requests on behalf of a user, rather than handling the requests in the browser.

=== Build menu items and web app links
The response from the `/api/rest/2.0/metadata/search` endpoint is a top-level array of objects. Each object contains the details needed to build menu items and links within the web application that load ThoughtSpot content.

The endpoint returns `metadata_name`, `metadata_id`, and `metadata_type` in each response item — enough information to build a simple menu with a link to the appropriate URL.

// MINOR-7: metadata_headers (plural) → metadata_header (singular) — confirmed from V2 response schema
// MINOR-6/CRIT-10: Removed V1 comparison text ("while the metadata/list V1 endpoint contains them
// in a slightly different structure") — V1 references removed
Within the `metadata_header` object returned for each item, the following properties are useful for building richer navigation UI:

* `description` — Text description added to the content by its creator
* `authorDisplayName` — Display name of the object creator or current owner
* `authorName` — Username of the object creator or current owner
* `created` — Object creation timestamp (milliseconds since epoch)
* `modified` — Last edit timestamp (milliseconds since epoch)
* `tags` — Array of tag objects, each with a `name` property

Individual visualizations on a Liveboard can be loaded using the `LiveboardEmbed` component by supplying both `liveboardId` and `vizId`.

The display of a visualization from a Liveboard differs from a saved Answer object, which is loaded via the `SearchEmbed` component. The saved Answer always displays the ThoughtSpot search bar and UI actions for editing, whereas visualizations display fewer UI elements and show the **More** menu for chart-level actions.

// Previous wording incorrectly implied it required a separate call "in the V1 REST API".
// In V2.0, setting include_visualization_headers=true returns viz headers inline in the
// same /metadata/search call — no separate call required.
To retrieve the list of all visualization details for a Liveboard in the same API call, set the `include_visualization_headers` request parameter to `true` (see the following section).

== Replicating the ThoughtSpot Liveboards or Answers page

// SOURCE: rest-api-v2-metadata-search.adoc response schema
The `/api/rest/2.0/metadata/search` endpoint provides the metadata required to replicate the Liveboards or Answers pages from the ThoughtSpot UI within your embedding application. See the link:https://github.com/thoughtspot/ts_everywhere_resources/blob/master/examples/menus_and_other_navigation_components/rest_api_ui_navigation_v2.html[V2.0 example on GitHub, window=_blank] for a complete reference implementation.

The following `type` values are supported:

* `LIVEBOARD` for Liveboards
* `ANSWER` for saved Answers
* `LOGICAL_TABLE` for data objects (Tables, Models, and Views)

For `LOGICAL_TABLE` objects, you can specify a `subtypes` filter to limit results to a specific type:

* `ONE_TO_ONE_LOGICAL` for Tables
* `WORKSHEET` for Models
* `AGGR_WORKSHEET` for Views

Additional filtering options include `record_size` and `record_offset` for pagination, and xref:rest-api-v2-metadata-search.adoc#_favorite_object_options[favorite options] to filter by favorites or creator.

The following example retrieves all Liveboards the signed-in user has access to and returns the results as a JavaScript array:

[source,javascript]
----
async function fetchMetadataList(type, options = {}) {
    const requestBody = {
        "metadata": [{ "type": type }],
        "record_offset": options.recordOffset ?? 0,
        "record_size": options.recordSize ?? 10000
    };

    // Add favorite filter if specified (MY = created by me, FAVOURITE = marked as favorite)
    if (options.filterAuthor) {
        requestBody["created_by_user_identifiers"] = [options.filterAuthor];
    }

    // Add sort options if specified
    if (options.sortBy) {
        requestBody["sort_options"] = {
            "field_name": options.sortBy,          // e.g. "NAME", "MODIFIED"
            "order": options.sortAsc ? "ASC" : "DESC"
        };
    }

    const response = await fetch('/api/rest/2.0/metadata/search', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'Accept': 'application/json'
        },
        body: JSON.stringify(requestBody),
        credentials: 'include'     // scopes request to the signed-in user
    });

    if (!response.ok) {
        throw new Error(`metadata/search request failed: ${response.status}`);
    }

    // V2.0 returns a top-level array directly (not nested under 'headers')
    return await response.json();
}

// Usage: fetch all Liveboards sorted by name
fetchMetadataList('LIVEBOARD', { sortBy: 'NAME', sortAsc: true })
    .then(results => renderNavigationFromResponse(results));
----

=== Rendering Liveboards or Answers pages similar to ThoughtSpot UI
// CRIT-8: Full rewrite. Previous version read V1 field names:
//   listResponse[i]['id'], ['name'], ['authorDisplayName'] — V1 shape
// V2.0 response shape: metadata_id, metadata_name at top level;
//   authorDisplayName inside metadata_header sub-object
// SOURCE: rest-api-v2-metadata-search.adoc response schema
If you want to render a listing similar to the Answers or Liveboards pages in the ThoughtSpot UI, your rendering function should read `metadata_id`, `metadata_name`, `metadata_header.tags`, `metadata_header.modified`, and `metadata_header.authorDisplayName` from each item in the response array:

[source,javascript]
----
function tableFromMetadataList(metadataSearchResults) {
    const table = document.createElement('table');

    // Table headers
    const thead = table.createTHead();
    const headerRow = thead.insertRow();
    ['Name', 'Tags', 'Modified', 'Author'].forEach(label => {
        const th = document.createElement('th');
        th.textContent = label;
        headerRow.appendChild(th);
    });

    // Table body
    const tbody = table.createTBody();
    metadataSearchResults.forEach(item => {
        const row = tbody.insertRow();

        // Name column — links to your embedding app's route for this content
        const nameCell = row.insertCell();
        const link = document.createElement('a');
        link.href = '#';
        link.textContent = item.metadata_name;
        link.addEventListener('click', () => loadContent(item.metadata_id));
        nameCell.appendChild(link);

        // Tags column — tags are inside metadata_header
        const tagsCell = row.insertCell();
        const tags = item.metadata_header?.tags ?? [];
        tagsCell.textContent = tags.map(t => t.name).join(', ');

        // Modified date column — timestamp in milliseconds
        const modifiedCell = row.insertCell();
        const modifiedDate = new Date(item.metadata_header?.modified ?? 0);
        modifiedCell.textContent = modifiedDate.toLocaleDateString();

        // Author column
        const authorCell = row.insertCell();
        authorCell.textContent = item.metadata_header?.authorDisplayName ?? '';

        tbody.appendChild(row);
    });

    return table;
}

// Usage: fetch Liveboards and render table
fetchMetadataList('LIVEBOARD', { sortBy: 'NAME', sortAsc: true })
    .then(results => tableFromMetadataList(results))
    .then(table => document.getElementById('main-content-div').appendChild(table));
----

Note that the `loadContent()` function in the name column anchor is a placeholder. Replace it with your application's routing logic for loading ThoughtSpot content.

=== Retrieve individual visualizations from a Liveboard

To retrieve the list of visualizations on a specific Liveboard, use the `/api/rest/2.0/metadata/search` endpoint with the `include_visualization_headers` parameter set to `true`. The visualization headers are returned inline in the same response — no separate API call is required.

[NOTE]
====
The V1 REST API endpoint `GET /tspublic/v1/metadata/listvizheaders` is deprecated. Use `/api/rest/2.0/metadata/search` with `include_visualization_headers: true` for all new implementations.
====

[source,javascript]
----
async function fetchLiveboardVizHeaders(liveboardGuid) {
    const requestBody = {
        "metadata": [
            {
                "type": "LIVEBOARD",
                "identifier": liveboardGuid
            }
        ],
        "include_visualization_headers": true
    };

    const response = await fetch('/api/rest/2.0/metadata/search', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'Accept': 'application/json'
        },
        body: JSON.stringify(requestBody),
        credentials: 'include'
    });

    if (!response.ok) {
        throw new Error(`metadata/search request failed: ${response.status}`);
    }

    const results = await response.json();

    // visualization_headers is returned per Liveboard item in the response array
    const vizHeaders = results[0]?.visualization_headers ?? [];
    return vizHeaders;
}

// Usage: build a list of viz links for a given Liveboard
fetchLiveboardVizHeaders('{liveboard-GUID}')
    .then(vizHeaders => {
        vizHeaders.forEach(viz => {
            console.log(`Viz ID: ${viz.id}, Name: ${viz.name}`);
            // Use viz.id with LiveboardEmbed to load individual visualizations:
            // new LiveboardEmbed('#embed', { liveboardId: '{liveboard-GUID}', vizId: viz.id })
        });
    });
----
